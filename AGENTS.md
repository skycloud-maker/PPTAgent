## 1. 이 문서의 목적

AI 에이전트(Claude 등)가 이 프로젝트의 코드를 작성할 때 반드시 따라야 할 규칙과 행동 방식을 정의한다.  
**코드를 작성하기 전에 반드시 이 문서를 먼저 읽어야 한다.**

---

## 2. 기본 원칙

### 2-1. 문서 우선
- 코드를 작성하기 전에 항상 관련 문서를 먼저 확인한다
- 문서와 코드가 충돌할 경우 **문서가 기준**이다
- 문서에 없는 내용을 임의로 구현하지 않는다 → 사용자에게 먼저 확인한다

### 2-2. 한 단위씩 실행
- PLANS.md에 정의된 실행 단위 하나만 수행한다
- 다음 단위를 미리 구현하지 않는다
- 단위 완료 후 반드시 사용자의 승인을 기다린다

### 2-3. 완료 기준 준수
- PLANS.md에 명시된 **완료 기준**을 모두 충족해야 해당 단위가 완료된 것이다
- 완료 기준 중 하나라도 미달 시 스스로 재시도한다
- 2회 이상 실패 시 사용자에게 보고하고 설계 재검토를 요청한다

### 2-4. 임의 판단 금지
- 설계 결정이 필요한 상황에서 임의로 판단하지 않는다
- "이렇게 해도 될 것 같다"는 판단으로 진행하지 않는다
- 불확실한 경우 반드시 사용자에게 질문 후 진행한다

---

## 3. 코드 작성 규칙

### 3-1. 언어 및 스타일
- 언어: Python 3.11+
- 포매터: `black` (줄 길이 88자)
- 린터: `ruff`
- 타입 힌트: 모든 함수에 필수
- docstring: 공개 함수/클래스에 필수 (한국어 허용)

### 3-2. 파일 구조 준수
- PLANS.md에 정의된 폴더 구조를 따른다
- 임의로 새 디렉토리나 파일을 추가하지 않는다
- 추가가 필요한 경우 사용자에게 먼저 확인한다

### 3-3. 환경변수
- API 키, 비밀값은 절대 코드에 하드코딩하지 않는다
- 항상 `.env` 파일과 `os.getenv()`를 사용한다
- 새로운 환경변수 추가 시 `.env.example`에도 반영한다

### 3-4. 에러 처리
- 모든 외부 API 호출(Claude API 등)은 try/except로 감싼다
- 사용자에게 보여지는 에러 메시지는 한국어로 작성한다
- 내부 로그 메시지는 영어로 작성한다
- 에러 발생 시 스택 트레이스를 로그에 기록한다

### 3-5. 보안 규칙 (SECURITY.md 참고)
- 생성된 `.pptx` 파일은 디스크에 영구 저장하지 않는다
- `tempfile` 모듈을 사용하고 완료 후 즉시 삭제한다
- 사용자 입력 내용을 로그에 기록하지 않는다
- Claude API로 전송하는 데이터를 로그에 기록하지 않는다

---

## 4. Streamlit 작성 규칙

- `st.session_state`를 적극 활용하여 단계 간 데이터를 유지한다
- 각 Step은 별도 함수로 분리한다: `render_step_1()`, `render_step_2()` 등
- 사용자 액션(버튼 클릭 등) 후 `st.rerun()`으로 상태를 갱신한다
- 로딩 중에는 항상 `st.spinner()` 또는 진행률 표시를 제공한다
- 에러 발생 시 `st.error()`로 표시하고 재시도 버튼을 제공한다

---

## 5. Slide Schema 규칙

- 모든 슬라이드 데이터는 반드시 `SlideSchema` (Pydantic 모델)를 통해 검증한다
- AI가 생성한 JSON을 그대로 렌더러에 전달하지 않는다
- 스키마 검증 실패 시 AI에게 재생성을 요청한다 (최대 3회)
- ARCHITECTURE.md에 정의된 슬라이드 타입 외의 타입은 사용하지 않는다

---

## 6. LLM 호출 규칙

- 항상 `LLMInterface`를 통해 호출한다 (ClaudeAdapter 직접 호출 금지)
- 프롬프트는 `core/prompts/` 디렉토리에 별도 파일로 관리한다
- 프롬프트를 코드 안에 인라인으로 작성하지 않는다
- API 호출 실패 시 최대 2회 재시도 후 사용자에게 에러를 표시한다
- 토큰 사용량을 로그에 기록한다 (내용은 기록하지 않음)

## 7. 커밋 메시지 규칙

```
[단위ID] 한 줄 요약

- 변경 내용 1
- 변경 내용 2
```

예시:
```
[1-A] 프로젝트 기본 구조 세팅

- 폴더 구조 생성 (core, templates, etc.)
- LLMInterface 추상 클래스 정의
- .env.example 작성
```

## 8. 작업 보고 형식

각 실행 단위 완료 후 사용자에게 아래 형식으로 보고한다:

```
## 완료 보고 — [단위ID]

**완료된 작업**
- 항목 1
- 항목 2

**완료 기준 체크**
- [x] 기준 1 충족
- [x] 기준 2 충족
- [ ] 기준 3 미충족 → 사유 설명

**확인 방법**
사용자가 직접 확인할 수 있는 방법 안내
예) `streamlit run app.py` 실행 후 브라우저에서 확인

**다음 단위 제안**
[단위ID] 설명
```

## 9. 하지 말아야 할 것 (Anti-patterns)

| 금지 행동 | 이유 |
|-----------|------|
| API 키 하드코딩 | 보안 위험 |
| 문서에 없는 기능 임의 추가 | 설계 일관성 파괴 |
| 여러 실행 단위 동시 진행 | 롤백 어려움 |
| 에러 무시 후 진행 | 잠재적 버그 누적 |
| 사용자 입력 로그 기록 | 보안 정책 위반 |
| .pptx 파일 서버 저장 | 보안 정책 위반 |
| LLMInterface 없이 직접 API 호출 | 추상화 레이어 파괴 |
| 완료 기준 미확인 후 다음 단위 진행 | 품질 기준 미달 누적 |
